lookahead: la=k 表示该 task 比同 progress 的 la=0 task 提前处理 batch index +k；la=0 = 处理本步 batch。
Gantt 中的红色虚线：跨线程 同 progress 数据/控制依赖中**导致 wait** 的那一条 binding 边（其它能被同线程顺序自动满足的不画）。覆盖 finish_shuffle@3 → start_input_dist@3 这种同 la slot 边、la=0 cluster 的 depends_on / slot 链（zero_grad/global_tokens/nccl_safety_barrier/forward → backward → finalize → opt → watchdog）、以及 prefetch=True 时的 prefetch_embeddings → backward（cross-la same_progress_sync）。在 default map 下只表现为 io→compute 的一条线；在 per_task 下能看到 forward→backward 把 backward 锁在 forward 之后。
Gantt task 宽度：粗略 CPU host-thread 时间估计（HSTU H100 单节点，prefetch=True，仅做相对比较；GPU 实际时间另算）。

📐 La cascade 推导：每对 +1 la 修的是哪个 host sync

每条链上的 task 都有"等前一个完成"的隐式 host sync。同 la → 同 iter → 等的对象 0ms 完成时间 → host 真等。+1 la 让 producer 跑在前一 iter，等的对象有整整 1 iter 在后台跑完，host 到时秒回。

红色箭头：跨 la 边（cascade 的关键），让 producer 跑在前一 iter，consumer 这一 iter 拿到的 result 已经在后台完成 → host wait 秒回。
灰色虚线：同 la 边（finish_shuffle@3 → start_input_dist@3），两个 task 在同 iter 同 batch 但跨 stream 跨 thread；如果在 4-thread map 上，host issue 是并发的、不阻塞；GPU 端用 wait_event 跨流同步。
注意：legacy 路径里 prefetch@1 → forward@0 已经是 cross-iter（这是 prefetch 模式的根源），cascade 没动这条；新增的 4 条 +1 la 是 (h2d↔start_shuffle, start_shuffle↔finish_shuffle, start_input_dist↔wait_input_dist, wait_input_dist↔prefetch)。

🔄 稳态 pipeline：6 batch 同时在飞

每行一条 la 链，每列一个 progress() iter。同列的 task 是 同一 iter 内并发触发的，处理 不同 batch（行号 = la 偏移）。稳态时 6 个 batch (Bk..Bk+5) 同时在不同位置流动，新 batch 从右侧入，最老的 batch 在 forward/backward 出。

稳态读法：竖看（同 iter）= 当前 progress() 触发的 6 个并发 task，作用于 6 个不同 batch（Bk 在 default，Bk+1 在 prefetch，..., Bk+5 在 h2d）。横看（同 la）= 同一条 chain 在连续 iter 上往前推一个 batch，一条 chain 内部完全是顺序的。
跨 la 同 batch 的关系：一个 batch Bk+5 从 iter k 的 h2d 入口进入 → iter k+1 的 start_shuffle → iter k+2 的 finish_shuffle / start_input_dist → iter k+3 的 wait_input_dist → iter k+4 的 prefetch → iter k+5 的 forward。**每跨一 iter 就跨 1 la**，正是 cascade 的全部含义。
资源利用：每个 iter 同时 6 路并发，理论上 default-stream forward + memcpy AllGather + data_dist NCCL + prefetch UVM 全部 overlap；实际能不能全 overlap 取决于 thread map（`default` map 把后 4 行挤在 compute 一条线程上 → 退化）。
📊 Benchmark observation (cw-dfw H100 ×8, job 11447285, 2026-04-30)
▸ 实测配置：上述 6-la cascade × HSTU_THREAD_MAP_VARIANT=default（2 threads: io / compute），nsys profile 50 iter，profile_step_start=150 profile_step_end=200
▸ parity：test_hstu_pipeline_matches_none_pipeline 通过 — bit-exact 跟 legacy JaggedMegatronTrainNonePipeline 一致 ✓
▸ la cascade 机制有效：wait_input_dist host block 从 ~8ms（cascade 之前）压到 0.4ms（splits a2a 已飞 1 整 iter，秒回）
▸ 但总吞吐没显著改善：avg6 TFLOPS = 1849 (la cascade) vs 1876 (round2 default 基线) ≈ -1.4% (噪声范围)
▸ 原因诊断：8ms host block 本来就在 critical path 之外（跟 default-stream forward/backward 并行）。真正的长杆是 (a) zero_grad 等上一 iter DDP grad allreduce 4.7ms idle、(b) shuffle/input_dist 的 NCCL 跟 default-stream compute 0% overlap（NCCL 跑在 default 流的 idle 窗口里，没真正藏到计算后面）
▸ 下一刀：要让 NCCL 完全藏住，必须先消除 default 流自身的 idle gap（zero_grad 等待 + Python launch overhead），然后 NCCL 才有机会跟连续 compute kernel overlap。
