lookahead: la=k 表示该 task 比同 progress 的 la=0 task 提前处理 batch index +k；la=0 = 处理本步 batch。
Gantt 中的红色虚线：跨线程同 progress 内**导致 wait** 的 binding 边。共 4 类边模型：
ord（slot/depends_on）— 只在 producer.la == consumer.la 时画 wait；不同 la 由 engine ring rotation 跨 progress 满足。
sync（same_progress_sync，如 prefetch_embeddings→backward）— 不分 la 永远画。
fifo（同 stream 不同线程）— engine 强制提交顺序（executor.py:310-322），不分 la 永远画。
nccl（_NcclOrderedLock 全局 ticket 链 executor.py:367-413）— 所有 nccl=True task 按 declaration order 串行 acquire/release，**critical section 是整个 task body**（含 host sync），不分 la / 不分 comm。这是 OLD la 和 NEW la 都有的 io→compute ~3ms gap 真正成因（ticket 链：start_shuffle→finish_shuffle→start_input_dist→wait_input_dist→global_tokens→backward→finalize）。**la cascade 没解决这个问题**——它只压了 host sync 时长，没缩 NCCL ticket 链。
Gantt task 宽度：粗略 CPU host-thread 时间估计（HSTU H100 单节点，prefetch=True，仅做相对比较；GPU 实际时间另算）。
📊 Round3 benchmark sweep (cw-dfw H100 ×8, 2026-04-30, sweep mtms_sweep_round3_20260430_140637)
▸ 9 variants，全部 nsys ON，每个 50 iter profile 区间，TFLOPS = avg(iter 599/699/799)。
▸ la cascade 机制本身有效：NVTX 上 wait_input_dist host block 从 ~8ms 压到 ~0.4ms（splits a2a 已飞 1 整 iter）
▸ 但 wall-clock 没收益（−0.1%）：nsys SQLite 量化（rank0 la_cascade.sqlite）：start_input_dist on data_dist stream 实际启动延迟 3-4ms，跟 OLD la new_default 一样。原因是 _NcclOrderedLock task 级 critical section（executor.py:367-413）：compute 线程要 acquire ticket 2 必须等 io 线程释放 ticket 1，而 ticket 1 包了整个 finish_shuffle body 含 KK background wait。la cascade 砍了 host sync 时长但没缩 NCCL ticket 链。
▸ 下一刀方向：把 NCCL lock 从 task 级降到 op 级（每个 dist.X() 单独 acquire/release），或加 release_early() API 让 task 在 NCCL queue 完后立刻释放，host sync 不持锁。
